1. Kết luận nhanh

Đây là backend quản trị quy trình sản xuất/xuất bản manga. Hệ thống xoay quanh project, editor board, cây folder/file, materials dạng JSON, tasks, vùng comment trên trang/khung và applications để gửi duyệt bản thảo hoặc yêu cầu xuất bản.

Backend dùng NestJS 11, Prisma 7 với PostgreSQL, JWT access token, refresh token lưu DB, Google OAuth, email verify/reset password qua Nodemailer, Swagger tại /docs và prefix API toàn cục /api.

Thiết kế phân quyền là phần quan trọng nhất: có quyền hệ thống theo role SYS và quyền theo project qua role PRJ. Ngoài role, hệ thống còn sinh quyền ngữ cảnh như project:owner, board:owner, board:leader, board:member.

3. Domain hệ thống

Người dùng và staff

User có email/password hoặc Google account. User thường tự đăng ký rồi verify email; staff/admin có thể tạo user active ngay và gán role hệ thống.

Role và permission

Role có scope SYS hoặc PRJ. Permission cũng có scope tương ứng. Role default được dùng khi đăng ký user hoặc tạo project.

Project

Project là đơn vị manga/chapter/công việc chính. Project có owner là người tạo, có members theo role project và có thể thuộc một editor board.

Editor board

Editor board gom editor/reviewer, có owner, members và tối đa một lead do API setToLead reset các lead khác về false.

Asset/work tree

Trong project có cây folder. Folder chứa file. File có materials JSON và tasks. Task có thể lồng nhau và có frame/comment để review theo vùng.

Application

Application là đơn gửi duyệt: MANUSCRIPT_REVIEW hoặc PUBLISH_REQUEST, có trạng thái PENDING, APPROVE, REJECT, CANCELLED.

4. Luồng flow nghiệp vụ

Backend này có thể đọc như một chuỗi flow nối tiếp nhau: trước hết user phải được xác thực, sau đó được kiểm quyền, rồi mới đi vào các workflow quản lý project, editor board, tài nguyên sản xuất manga, review/comment và gửi duyệt/xuất bản.

4.1. Flow đăng ký, xác thực và duy trì phiên

  • Đăng ký tài khoản: client gọi POST /api/auth/register với email, password và display name. Backend chuẩn hóa email về lowercase, hash password bằng bcrypt, tạo user chưa active, gán role hệ thống mặc định SYS, sinh token verify email và gửi email xác thực.
  • Xác thực email: user mở link hoặc client gửi token đến POST /api/auth/verify-email. Nếu token còn hạn, backend bật isActive=true và xóa token verify khỏi user.
  • Đăng nhập: client gọi POST /api/auth/login. Backend kiểm tra user tồn tại, có password, đã active và password đúng. Kết quả là access token trả trong body, refresh token được set vào cookie refreshToken dạng httpOnly.
  • Duy trì phiên: khi access token hết hạn, client gọi POST /api/auth/refresh. Backend lấy refresh token từ cookie hoặc body, kiểm tra token trong DB, verify JWT, tạo access token mới, tạo refresh token mới và xóa refresh token cũ. Đây là cơ chế refresh token rotation.
  • Đăng xuất: client gọi POST /api/auth/logout. Backend xóa refresh token khỏi DB, clear cookie và nếu request có Bearer access token thì đưa access token đó vào blacklist tới thời điểm hết hạn.
  • Quên/đặt lại mật khẩu: POST /api/auth/forgot tạo password reset token và gửi email; POST /api/auth/reset đổi password, xóa reset token và xóa toàn bộ refresh token của user để bắt đăng nhập lại.
  • Google OAuth: GET /api/auth/google chuyển qua Google; callback tìm user bằng Google id hoặc email, tự link Google nếu email đã có, hoặc tạo user mới active ngay. Sau đó backend redirect về frontend với access token và set refresh cookie.

4.2. Flow kiểm quyền cho mọi request

  • Bước 1 - global auth: mọi endpoint mặc định đi qua GlobalAuthGuard. Chỉ endpoint có @Public() mới bỏ qua JWT.
  • Bước 2 - kiểm access token: AccessTokenStrategy lấy Bearer token, kiểm token có nằm trong blacklist_tokens không, sau đó kiểm user trong DB còn tồn tại.
  • Bước 3 - đọc metadata permission: PermissionGuard đọc @Permissions trên controller method. Nếu method không khai báo permission thì chỉ cần JWT là qua.
  • Bước 4 - gom quyền: nếu là quyền hệ thống, backend query từ user_roles -> role_permissions -> permissions. Nếu là quyền tài nguyên, backend truy ngược resource về project/board để xác định user có phải member, owner hoặc board lead không.
  • Bước 5 - so khớp: mode ANY nghĩa là user chỉ cần có một quyền trong danh sách; mode ALL yêu cầu đủ toàn bộ quyền.
Cách hiểu nhanh: role SYS trả lời câu hỏi "user này có quyền quản trị toàn hệ thống không?", còn role PRJ trả lời câu hỏi "user này làm được gì trong project cụ thể này?". Các quyền như project:owner và board:leader không đến trực tiếp từ bảng permission, mà được suy ra từ quan hệ dữ liệu.

4.3. Flow tạo và vận hành project

  • Tạo project: user gọi POST /api/projects. Backend tạo project, lấy role project mặc định PRJ, rồi tạo bản ghi UserProject để creator trở thành member của project đó.
  • Xem danh sách project: GET /api/projects trả về project user tạo hoặc project user đang là member. Query me=true thu hẹp về project user tạo.
  • Quản lý thông tin project: owner hoặc user có quyền project:update có thể sửa project; owner hoặc project:delete có thể xóa project.
  • Quản lý member: owner hoặc quyền member tương ứng có thể list, add, update role hoặc remove member. Khi add/update member, backend chỉ chấp nhận role scope PRJ.
  • Bảo vệ owner: backend không cho remove project owner khỏi project qua API remove member.
  • Gắn editor board: project có thể được set editor board bằng POST /api/projects/:id/editor-boards, hoặc được board kéo vào bằng POST /api/editor-boards/:id/projects.

4.4. Flow editor board

  • Tạo board: user gọi POST /api/editor-boards. Backend tạo board và thêm creator vào UserEditorBoard.
  • Quyền board: creator được xem là board:owner do createdBy; member có board:member; một member có thể được đặt làm board:leader.
  • Thêm member: owner gọi POST /api/editor-boards/:id/members để thêm user vào board. API dùng skipDuplicates nên thêm trùng không làm vỡ flow.
  • Set lead: owner gọi PATCH /api/editor-boards/:id/members/:userId/lead. Backend reset toàn bộ isLead=false trong board rồi set user được chọn thành lead, tức mỗi board chỉ có một lead theo logic service.
  • Gắn project vào board: owner hoặc lead gọi POST /api/editor-boards/:id/projects. Backend chặn project đã thuộc board khác, nhưng cho phép project chưa có board hoặc đã thuộc chính board đó.
  • Xem application của board: board xem các application type PUBLISH_REQUEST từ project thuộc board qua GET /api/editor-boards/:id/applications.

4.5. Flow cây nội dung sản xuất manga

  • Project chứa folder: owner hoặc user có project:folder.create tạo folder gốc bằng POST /api/projects/:id/folders. Folder luôn thuộc một project.
  • Folder có cây con: từ một folder, client gọi POST /api/folders/:id/children để tạo folder con. Backend tự lấy projectId từ parent, bảo đảm child folder nằm cùng project.
  • Folder chứa file: client gọi POST /api/folders/:id/files để tạo file. File có title/description và thuộc folder.
  • File có materials: client gọi POST /api/files/:id/materials để lưu JSON materials. Hệ thống không áp schema cứng cho JSON này, nên mobile/frontend có thể lưu cấu trúc như layers, image references, page metadata hoặc editor state tùy nghiệp vụ.
  • File có tasks: client gọi POST /api/files/:id/tasks để tạo task cho file. Task có status PENDING, INPROGRESS, REVIEW, DONE, có thể có parent task và người được assign qua assignedBy.
  • Task có task con: GET /api/tasks/:id/children đọc task con theo parentId. Điều này cho phép chia một việc lớn thành nhiều việc nhỏ theo trang, panel hoặc bước xử lý.

4.6. Flow review bằng frame và comment

  • Tạo vùng cần comment: client gọi POST /api/tasks/:id/frames với startX, startY, endX, endY. Đây là vùng tọa độ trên task/file để reviewer chỉ đúng chỗ cần góp ý.
  • Comment trong frame: client gọi POST /api/frames/:id/comments với content dạng JSON. Vì content là JSON, frontend có thể lưu text thuần, rich text, mention, attachment hoặc trạng thái resolve nếu sau này mở rộng.
  • Xem comment theo frame: GET /api/frames/:id/comments trả comment trong một vùng cụ thể.
  • Xem comment theo task: GET /api/tasks/:id/comments gom tất cả frame của task, sau đó list toàn bộ comment thuộc các frame đó. Đây là endpoint tiện cho màn hình task detail hoặc review summary.
  • Cập nhật/xóa frame: PATCH /api/frames/:id sửa tọa độ; DELETE /api/frames/:id xóa frame. Do schema cascade comment theo frame, xóa frame cũng kéo theo comments liên quan.

4.7. Flow application review và publish request

  • Tạo application trong project: owner hoặc user có quyền update project gọi POST /api/projects/:id/applications. Application gồm title, description, materials JSON và type MANUSCRIPT_REVIEW hoặc PUBLISH_REQUEST.
  • List application theo project: GET /api/projects/:id/applications hỗ trợ search, filter type/status và phân trang.
  • Board xem publish request: GET /api/editor-boards/:id/applications chỉ lấy application type PUBLISH_REQUEST từ các project thuộc board.
  • Cập nhật application: user có project:application.update hoặc board leader có thể sửa title, description, materials.
  • Duyệt/từ chối/hủy: user có project:application.approve hoặc board leader gọi PATCH /api/applications/:id/status để đổi status sang APPROVE, REJECT hoặc CANCELLED. Backend ghi người duyệt vào verifyBy.

4.8. Flow project stats

  • Import stats theo project: client gọi POST /api/projects/:id/stats với metrics dạng JSON. Backend tìm stat hiện có của project; nếu có thì update, chưa có thì create.
  • Đọc stats theo project: GET /api/projects/:id/stats trả metrics hiện tại của project.
  • CRUD trực tiếp: module project-stats cũng có GET/PATCH/DELETE /api/project-stats/:id để thao tác theo id bản ghi stat.
  • Ý nghĩa nghiệp vụ: metrics là JSON tự do, phù hợp lưu số trang, tiến độ, số task theo trạng thái, dữ liệu dashboard hoặc dữ liệu import từ công cụ khác.

4.9. Flow tổng hợp từ góc nhìn mobile/frontend

  • Mở app: app kiểm access token hiện có; nếu hết hạn hoặc mất access token thì gọi refresh bằng cookie/body refresh token.
  • Sau login: app gọi /api/users/me để lấy profile, có thể gọi /api/permissions/me/sys để biết quyền hệ thống.
  • Màn hình project: app gọi /api/projects, sau đó vào project detail bằng /api/projects/:id, đồng thời có thể gọi /api/permissions/me/projects/:id để bật/tắt nút theo quyền.
  • Màn hình nội dung project: app đọc folders bằng /api/projects/:id/folders, đi sâu qua /api/folders/:id/children và /api/folders/:id/files, rồi đọc materials/tasks của file.
  • Màn hình review: app đọc task, frames và comments; user tạo frame/comment khi có quyền tương ứng.
  • Màn hình board: app list boards, vào board detail, xem members/projects/applications. Lead hoặc owner có thêm hành động gắn project và xử lý publish requests.

6. Xác thực

Luồng email/password

  • Register: chuẩn hóa email lowercase, hash password bằng bcrypt, tạo user isActive=false, gán role SYS default, tạo token verify email, gửi mail.
  • Verify email: nhận token, kiểm tra token và hạn, bật isActive=true, xóa token verify.
  • Login: kiểm tra user có password, active và password đúng; trả access token và set refresh token vào cookie refreshToken.
  • Refresh: nhận refresh token từ cookie hoặc body, kiểm tra tồn tại trong DB, kiểm JWT, tạo access token mới, rotate refresh token và xóa refresh cũ.
  • Logout: nếu có access token thì upsert vào blacklist_tokens tới lúc hết hạn; xóa refresh token và clear cookie.
  • Reset password: forgot tạo token reset; reset hash password mới và xóa toàn bộ refresh token của user.

Luồng Google

  • GET /api/auth/google chuyển user sang Google OAuth.
  • Callback tìm user bằng googleId hoặc email. Nếu email đã có nhưng chưa link Google, backend link Google vào user đó và active user.
  • Nếu chưa có user, backend tạo user active, không có password, gán role SYS default.
  • Callback redirect frontend về /auth/oauth-success?access_token=... và set refresh cookie.
  • User đã login có thể link Google qua /api/users/me/link-account; state được ký bằng access token secret.
Cookie refresh token: cookie dùng httpOnly, sameSite=none, path /api/auth. Trường secure phụ thuộc NODE_ENV === production, nhưng code gọi requireEnv('NODE_ENV'), nghĩa là thiếu NODE_ENV có thể làm app lỗi khi load controller.

7. Phân quyền

Cơ chế guard

  • @Public(): bỏ qua global auth và permission guard.
  • GlobalAuthGuard: mặc định mọi endpoint cần JWT access token nếu không public.
  • AccessTokenStrategy: lấy Bearer token, kiểm token blacklist, gọi UsersService.findOne(userId) để bảo đảm user tồn tại.
  • PermissionGuard: đọc metadata @Permissions. Nếu mode ANY, chỉ cần một permission; nếu ALL, phải có tất cả.

8. Module nghiệp vụ

AuthModule

Đăng ký, verify email, login, refresh, logout, forgot/reset password, Google OAuth. Phụ thuộc MailService, PrismaService, JwtService.

UsersModule

Profile current user, password change, admin/staff user management, gán SYS roles, truy vấn projects/editor boards của user, link Google account.

RolesModule

CRUD role, filter theo scope, đảm bảo mỗi scope chỉ một default role, không xóa role đang được user hoặc project dùng.

PermissionsModule

List/filter/update permission metadata, trả quyền hiện tại của user ở mức SYS, project hoặc board.

ProjectsModule

CRUD project, member management, set editor board, list/create applications, list/create folders, read/import project stats.

EditorBoardsModule

CRUD editor board, member management, set lead, gắn projects vào board, xem projects/applications của board.

FoldersModule

CRUD folder, tạo child folder, list/create file trong folder.

FilesModule

CRUD file, lấy/tạo materials JSON cho file, list/create tasks cho file.

TasksModule

CRUD task, list child tasks, list/create frames, aggregate comments của mọi frame trong task.

FramesModule

CRUD frame tọa độ, list/create comments trong frame.

ApplicationsModule

List/detail/update/delete application, update status và ghi người verify.

ProjectStatsModule

CRUD trực tiếp theo id của project stat; module Projects cũng có endpoint get/import stats theo project id.

11. Điểm cần chú ý

PermissionGuard có giả định rất mạnh về :id

PermissionGuard luôn lấy request.params['id'] làm id của resource trong metadata. Điều này đúng với endpoint như /projects/:id, /files/:id, /tasks/:id. Nhưng nó dễ sai với endpoint lồng nhau, ví dụ /folders/:id/files khai báo resource FILE trong khi :id thực tế là folder id.

Các endpoint có rủi ro tương tự: tạo/list file trong folder, materials/tasks trong file, frames/comments trong task, comments trong frame, list applications không có :id, và /project-stats/:id dùng project stat id nhưng khai báo resource PROJECT. Nên cân nhắc metadata cho param name hoặc permission helper theo parent resource.

Collection endpoint với resource nhưng không có id

GET /applications có @Permissions(... resource: 'APPLICATION') nhưng không có path param :id. Guard sẽ truyền NaN vào logic resource. Điều này có thể dẫn tới lỗi runtime thay vì trả 403/200 như mong muốn.

Route test e2e có thể cũ

E2E test hiện kỳ vọng GET / trả Hello World!. Trong app thật có app.setGlobalPrefix('api'), nên nếu test không set cùng prefix thì test này có thể không phản ánh runtime thực tế.

ProjectStat import là upsert logic nhưng schema chưa unique

ProjectsService.importProjectStats tìm findFirst({ projectId }), update nếu có, create nếu chưa có. Schema chỉ index projectId, chưa unique. Nếu dữ liệu từng có duplicate project stats, logic sẽ chỉ update bản ghi đầu tiên.

Pattern code ổn định

Service nhìn chung nhất quán: dùng Logger, handleError, ensureEntity, transaction cho thao tác nhiều bảng, pagination helper, Prisma select để kiểm soát response. Đây là nền tảng tốt để frontend/mobile dựa vào.
